Overview

이 대시보드가 하는 일

① 세션 시각화

Claude Code 훅 이벤트를 받아 픽셀아트 마스코트로 세션 상태(idle / thinking / running / compacting / ended)를 실시간 표시.

② 사용량 & 비용

~/.claude/projects/**.jsonl을 SQLite에 증분 적재해 일별·모델별·프로젝트별 토큰과 캐시 효율을 집계.

③ 커뮤니티

MySQL 백엔드 위에 회원가입·그룹·초대 코드·그룹 대시보드 SSE까지 옵션으로 제공.

아키텍처 한 장 요약

Claude Code  ──(hook)──▶  hook-handler.sh  ──HTTP──▶  server.js
                                                           │
                                                           ├─ /api/stream        (세션 SSE)
                                                           ├─ /api/usage/stream  (사용량 SSE)
                                                           ├─ data/session_*.json (세션 영속화)
                                                           ├─ data/usage.db        (JSONL → SQLite)

핵심 포인트

  • setup-hooks.sh로 settings.json에 한 번에 머지 설치 (기존 훅·env·permissions 보존, .bak 백업)

Design Decision

왜 로그 파싱인가? — vs Claude OpenTelemetry

✓ 로그 파일 파싱 (채택)

  • 추가 환경변수 설정 없이 바로 사용
  • 이미 Claude Code가 남기는 로컬 데이터만으로 구현 가능
  • 원하는 정보를 원하는 대로 가공

✗ 한계

  • 로그 쓰기 자체가 실시간이 아님 (세션 저장/종료 시점에 적힘)
  • 완전 실시간이 아닌, "로그 append 후 반영" 구조

✓ OpenTelemetry로 얻는 것

  • usage 이벤트를 더 직접·빠르게 수신
  • 로그 쓰기 지연 우회

✗ 단점

  • 사용자가 직접 환경변수 설정 필요 (OTEL_EXPORTER_OTLP_ENDPOINT)
  • 대시보드 미기동 시 데이터 유실
  • "로컬에서 바로 켜서 쓴다"는 목표와 거리가 있음

👉 우선 로그 파일 기반으로 채택. 실시간성이 더 중요해지면 OTel로 교체할 수 있도록 확장 가능성은 열어둠.

Design Decision

같은 파일 동시 읽기 문제

🐛 문제 상황

  1. watcher 이벤트 A가 들어와 parseFile() 시작
  2. A가 끝나기 전 같은 파일에 이벤트 B가 또 들어옴
  3. B도 같은 last_offset를 읽음
  4. A·B가 같은 append 구간을 동시에 파싱
  5. 토큰·usage 레코드 중복 집계 발생

DB 쓰기는 동기지만, 파일 읽기는 비동기 스트림이라 같은 파일 파서가 동시에 둘 이상 돌 수 있음.

✓ 해결: 파일별 직렬 큐

  • 다른 파일은 병렬로 처리
  • 같은 파일만 순차 처리
  • 파일 경로를 키로 하는 Promise Chain 큐
const fileParseQueues = new Map();

function enqueueFileParse(filePath, sessionId, fallbackProjectPath) {
  const key = path.resolve(filePath);
  const previous = fileParseQueues.get(key) || Promise.resolve();
  const current = previous
    .catch(() => {})
    .then(async () => {
      await parseFile(filePath, sessionId, fallbackProjectPath);
      broadcastUsage();
    });
  fileParseQueues.set(key, current);
  current.finally(() => {
    if (fileParseQueues.get(key) === current) {
      fileParseQueues.delete(key);
    }
  });
  return current;
}

Feature · Group Chat

그룹 채팅 (SSE × MySQL)

그룹 단위 채팅은 routes/chat.js가 전담합니다. 실시간 전송은 SSE, 영속화는 MySQL. 인증·멤버십은 community와 동일한 세션 쿠키로 검증.

전송 흐름

1
구독GET /api/chat/groups/:groupId/stream — 멤버십 체크 후 SSE 오픈, 최근 50개 메시지를 init 이벤트로 푸시
2
전송POST /api/chat/groups/:groupId/messages — messages 테이블에 INSERT 후 같은 그룹 SSE 구독자에게 chat 이벤트로 브로드캐스트
3
읽음 처리POST /api/chat/groups/:groupId/read — group_member_reads의 last_read_message_id 갱신 (GREATEST로 역행 방지)
4
안 읽음 배지GET /api/chat/unread — 그룹별 unreadCount 일괄 반환 (카카오톡 스타일 뱃지)

실시간 이벤트 종류

connectedSSE 연결 직후
init최근 50개 메시지 일괄
chat새 메시지 도착
presence접속 중인 멤버 스냅샷 (입·퇴장 시 갱신)
member_change실제 그룹 가입·탈퇴 (창 토글과 구분)

제약 / 운영

  • 메시지 최대 2,000자
  • history 기본 50 / 최대 200 (?limit=)
  • 30초 keep-alive ping
  • 로그아웃 시 kickMember() — 모든 그룹 SSE 강제 종료 → presence에서 즉시 사라짐
  • FK 미사용: 참조 테이블 엔진 차이로 인한 실패 회피, 무결성은 requireGroupMembership으로 앱 레이어에서 보장
